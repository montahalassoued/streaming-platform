# Frontend Gaps

This document lists the screens and flows that are blocked, partially working, or require workarounds due to known backend limitations. Each entry names the affected screen, describes the gap, and states what the frontend should do until the gap is resolved.

---

## 1. Email Delivery Not Wired to a Real Provider

**Affects:** Verify Email screen, Forgot Password screen, Reset Password screen

**Gap:** The backend persists verification and password-reset tokens correctly, but the email sending integration uses a no-op/mock transport by default. In development, no email is actually delivered to the user's inbox.

**Workaround for development:**
- Check the server logs — the token value is logged when `NODE_ENV` is not `production`.
- Manually copy the token from the logs and append it to the URL: `/verify-email?token=<value>` or `/reset-password?token=<value>`.

**What the frontend should not do:** Assume the email will arrive in production without confirming the email provider is configured. The `EMAIL_*` environment variables must be set before going live.

---

## 2. Donation Checkout URL Is a Mock

**Affects:** Donate screen / donation modal

**Gap:** `POST /donations` returns a `checkoutUrl` but this URL is a placeholder generated by `MockPaymentProvider`. It does not redirect to a real payment page.

**Workaround for development:**
- Test the form submission and the `POST /donations/webhook` endpoint manually using a tool like Postman or curl to simulate `payment.succeeded`.
- Do not build the "payment return" UX yet — it has no real behavior to test against.

**What the frontend should do when the real provider is wired:**
- Redirect the user to `checkoutUrl` (new tab or same tab depending on UX preference).
- Handle the provider's return redirect URL with a success/failure query param.
- Optionally poll `GET /donations/:id` to confirm status before showing a success message.

---

## 3. Subscription Has No Payment Step

**Affects:** Subscribe button / subscription modal

**Gap:** `POST /subscriptions` creates a subscription record immediately without any payment gate. There is no price, no checkout, and no billing cycle enforcement beyond the 30-day expiry field.

**Workaround for development:**
- Build the subscribe/unsubscribe UI as if it is a free action.
- Display the 30-day expiry date returned in the subscription record.

**What the frontend should expect when billing is added:**
- The `POST /subscriptions` response will likely change to include a `checkoutUrl` similar to donations.
- A `price` or `tier` field may be added to the response.

---

## 4. SSE Does Not Work Across Multiple Backend Replicas

**Affects:** Notifications (bell icon / tray), stream-went-live toasts

**Gap:** SSE connections are stored in an in-memory `Map` per process. If the backend runs behind a load balancer with multiple replicas, a follower connected to replica A will miss events published by replica B.

**Workaround:**
- In development (single process) this is not an issue.
- In a multi-replica deployment, configure sticky sessions on the load balancer (e.g., nginx `ip_hash`).

**What the frontend should do:**
- Implement EventSource reconnect logic (native auto-reconnect is usually sufficient).
- Do not rely on SSE for critical delivery — treat it as best-effort and supplement with polling if necessary.

---

## 5. Admin Panel Has No Moderation Routes

**Affects:** Admin Panel screen

**Gap:** The `/admin` module only exposes user listing, promote, and demote. There are no routes for:
- Banning / unbanning users
- Removing streams or VODs
- Reviewing reported content
- Category management from the admin panel (categories have their own admin-guarded routes at `/categories`)

**Workaround:**
- Build only the user listing and promote/demote UI for now.
- Leave placeholders for moderation tabs so they can be filled in when the routes exist.

---

## 6. Chat Reactions Have No API

**Affects:** Chat message reaction UI (likes, emoji reactions)

**Gap:** `ChatMessageReactionEntity` exists in the database schema (table `chat_message_reactions`) but no API endpoints expose creating, reading, or deleting reactions.

**Workaround:**
- Do not build reaction UI until the endpoints are added.
- The schema supports `userId`, `messageId`, and `reaction` (string like `"like"`, `"heart"`).

---

## 7. VOD Creation Is Automatic — No Upload Flow

**Affects:** VOD management in the streamer dashboard

**Gap:** VODs are created automatically by `VodProcessWorker` when a stream ends. `POST /vods` exists but it expects a `videoUrl` (an already-hosted HLS URL). The backend has no file upload endpoint.

**Workaround:**
- Do not build a "upload VOD" flow.
- Show VODs that appear automatically after a stream ends.
- If manual VOD management is needed (edit title, toggle visibility), use `PATCH /vods/:id`.

---

## 8. Streamer Settings Update Has No Frontend Push

**Affects:** Any screen that displays channel settings in real time

**Gap:** When a streamer updates settings, the backend publishes `streamer.settings.updated` on Redis, but `NotificationsService` intentionally does not push this to the frontend (the SSE client map is keyed by `userId`, not `streamerId`, and the mapping requires an extra DB lookup with an undefined frontend contract).

**Workaround:**
- Refetch `GET /streamer/me` or `GET /users/:username` after a settings update rather than expecting a push event.

---

## 9. No Rate Limiting

**Affects:** Register, Login, Forgot Password, Donate, and other write endpoints

**Gap:** The backend has no rate limiting or throttling on any endpoint.

**What the frontend should do:**
- Implement client-side debounce on form submissions.
- Be aware that in production, multiple rapid submissions from the same user are not blocked server-side.
